# codingverse v1 实施计划

> 目标：落 SQLite 索引底座 + BM25/co-location 双路检索 + MCP server，打通三模态的第二模「检索」。
> 前置：MVP（M0–M7）已完成，63 tests pass，`cv pack` / `cv expand` 可用。
> 本计划是 v1 的**单一事实来源**，实施按本文里程碑顺序执行。

---

## 一、v1 范围边界

### 做
1. **Stage ③ SQLite 索引**（`indexdb/` 四模块：db / store / resolve / search）
2. **Engine 接线**：`index()` 落库、`search()` 真实检索、`close()` 关连接
3. **检索管线**：BM25（FTS5）+ co-location 图谱路 → 双路 RRF 融合
4. **CLI `cv search`**：从 stub 变真实命令
5. **pack 读索引**：有索引走 SQLite，无索引走现有内存管线（兼容）
6. **`@codingverse/mcp` 包**：暴露 search / pack / expand / get_file tools

### 不做（明确推迟）
| 留给 | 内容 | 代码标记 |
|------|------|---------|
| v2 | 调用图 callers/callees、impact 遍历、PageRank、SCIP | `Engine.callers/impact` stub |
| v2.5 | Dashboard 完整面板 | `Engine.stats` stub |
| v1.5+ | embedding 向量路（binarized 或真向量） | `chunks.embedding_tokens` 列先建好留空 |

### 两条已定决策
1. **不上向量**：v1 检索 = BM25（FTS5）+ co-location（同文件/同 scope 邻居），双路 RRF。`embed.ts` 暂不写，`chunks.embedding_tokens` 列保留但不写入。
2. **node:sqlite**：用 Node 22+ 内置 `node:sqlite`（`DatabaseSync`），不引 better-sqlite3，零原生编译。

## 二、关键技术决策（衍生于已定两条）

| 决策 | 选择 | 理由 / 验证 |
|------|------|------------|
| SQLite 驱动 | `node:sqlite` `DatabaseSync` | 已验证：Node v24.14.0 无需 flag，FTS5 可用（SQLite 3.51.2）。仅 `ExperimentalWarning`，可 `process.removeAllListeners('warning')` 或 `--no-warnings` 抑制 |
| 库文件位置 | `<repo>/.codingverse/index.db` | 与现有 `parse-cache.json` / `token-cache.json` / `expand-map.json` 同目录 |
| 同步 API | `DatabaseSync` 全同步 | node:sqlite 就是同步 API；Engine 方法保持 async 签名（返回 Promise）以兼容未来异步驱动 |
| 检索路数 | 双路（BM25 + co-location） | 原 design.md 三路里的 vector 路推迟；graph 路用 co-location 启发式顶替真调用图 |
| RRF 公式 | `score = Σ 1/(k + rank_i)`，k=60 | 抄 Tabby `code.rs:86-162` |
| co-location 定义 | 同文件 + 同 scope 嵌套 + 相邻行窗口 | 抄 CodeGraph `context/index.ts` co-location 启发式，无调用图时的图路近似 |
| FTS5 触发器 | external-content + AFTER INSERT/UPDATE/DELETE 触发器 | 抄 CodeGraph schema.sql:97-123，避免正文冗余存储 |
| 引用解析 | 二阶段：parse 产出 unresolved_refs → resolve 按名匹配 nodes | 抄 CodeGraph；v1 只做同文件/同语言名匹配，不做跨文件类型推断（留 SCIP） |
| MCP 传输 | stdio（默认）+ 可选 HTTP | stdio 是 MCP 主流，Claude Code/Cursor 默认 stdio |
| node:sqlite 类型 | `@types/node` 已含（v22+） | 已在 devDependencies，无需新增 |

### V1-1 db.ts：连接 + migration

**目标**：封装 node:sqlite 连接，提供 `migrate()` 跑 schema.sql。

**文件**：`packages/core/src/indexdb/db.ts`

**API 契约**：
```typescript
import { DatabaseSync } from "node:sqlite";

export interface DbOptions {
  /** 覆盖默认路径；测试用 ':memory:'。 */
  dbPath?: string;
  /** 抑制 ExperimentalWarning（默认 true）。 */
  suppressWarning?: boolean;
}

export class IndexDb {
  readonly db: DatabaseSync;
  constructor(opts?: DbOptions);
  /** 跑 schema.sql（幂等，IF NOT EXISTS）。返回已建表数。 */
  migrate(): number;
  /** 预编译常用 statement（性能）。在 migrate 后调。 */
  prepareStatements(): void;
  close(): void;
}
```

**实施细节**：
- `migrate()`：用 `fs.readFileSync` 读 `@codingverse/shared` 的 `schema.sql`（经 `require.resolve`），`db.exec()` 整体执行。schema 全用 `IF NOT EXISTS`，幂等。
- `prepareStatements()`：预编译高频 statement（insertNode / insertChunk / insertEdge / upsertFile / ftsSearch 等），存到 `this.statements` Map。
- 警告抑制：构造时 `const orig = process.emitWarning; ...` ��更简单 `process.removeAllListeners('warning')` 一次性。**只在 IndexDb 构造时做一次**，不影响全局其他警告行为。推荐用 `process.on('warning', (w) => { if (w.name === 'ExperimentalWarning' && w.message.includes('SQLite')) return; ... })` 的方式，精准过滤。
- WAL 模式：`db.exec("PRAGMA journal_mode=WAL; PRAGMA synchronous=NORMAL;")` 提升并发写。

**测试**（`indexdb/db.test.ts`）：
- `:memory:` 建库后 `table_list` 含 nodes/edges/chunks/files/unresolved_refs/nodes_fts/chunks_fts/meta
- 重复 `migrate()` 幂等（不报错、表数不变）
- WAL 设置生效（`PRAGMA journal_mode` 返回 'wal'，但 `:memory:` 下可能是 memory——用临时文件测）

### V1-2 store.ts：写入层

**目标**：把 `ParsedFile[]` 写进 SQLite nodes/edges/chunks/files + FTS5。

**文件**：`packages/core/src/indexdb/store.ts`

**API 契约**：
```typescript
export interface StoreInput {
  parsed: ParsedFile[];
  sources: Map<string, string>; // path → content（算 gitBlobHash 用）
}

export interface StoreStats {
  nodes: number;
  edges: number;
  chunks: number;
  files: number;
  unresolved: number;
}

export class IndexStore {
  constructor(db: IndexDb);
  /** 全量写入：先清旧（按 file 批次），再插新。单事务。 */
  write(input: StoreInput): StoreStats;
  /** 增量写入：只处理 changedPaths。未变的 file 跳过。 */
  writeIncremental(input: StoreInput, changedPaths: Set<string>): StoreStats;
  /** 删除已不存在的文件相关行（nodes/edges/chunks/files 级联）。 */
  pruneFiles(livePaths: Set<string>): number;
}
```

**实施细节**：
- **symbolId / chunkId**：复用现有 `symbolId(path, qualifiedName)`（skeleton.ts:19）和 `chunkId(path, startByte)`（chunker.ts:16）。**注意**：store 里要 export 这两个 id 函数到一个共用位置，避免循环依赖。建议放 `indexdb/ids.ts`，skeleton/chunker 改为从那 re-export。
- **事务**：`db.exec("BEGIN")` ... `db.exec("COMMIT")`，异常 `ROLLBACK`。node:sqlite 的 `DatabaseSync` 不支持 `.transaction()` 高级 API（better-sqlite3 才有），手动 BEGIN/COMMIT。
- **files 表**：`gitBlobHash` 复用 `cache/incremental.ts:gitBlobHash`，`parse_status` = `degraded ? 'degraded' : 'ok'`，`indexed_at = Date.now()`。
- **FTS5 写入**：
  - 若 V1-0 决定 external-content：不手动写 fts 表，靠触发器自动同步。
  - 若回退独立 FTS5 表：手动 `INSERT INTO nodes_fts VALUES (...)` / `chunks_fts`。
- **prepared statements**：用 V1-1 预编译的，`stmt.run(...)` 循环。node:sqlite 的 `StatementSync.run` 接受位置参数或 `.run({ named: val })`。
- **增量逻辑**：`writeIncremental` 对每个 changedPath 先 `DELETE FROM nodes WHERE file_path=?`（FK ON DELETE CASCADE 自动清 edges），再插入。未 changed 的 file 完全不碰。
- **unresolved_refs**：parse 产出的 `refs` 全部先写 `unresolved_refs`（含 `from_node_id` = 该 ref 所在的 symbol id，需按行号反查所属 symbol），V1-3 再解析成 edges。

**ref 所在 symbol 反查**：对一个 file 的 symbols 按行排序，每个 ref 按行找到 `startLine ≤ refLine ≤ endLine` 的最内层 symbol，取其 id 作 `from_node_id`。无所属则 `from_node_id = NULL`。

**测试**（`indexdb/store.test.ts`）：
- 写入 2 个 TS 文件后，nodes/chunks/files 行数正确
- FTS5 能查到（`SELECT body FROM chunks_fts WHERE chunks_fts MATCH 'format'`）
- `pruneFiles` 删一个文件后 nodes/edges/chunks 级联清空
- `writeIncremental` 只改一个文件时，另一文件的 nodes 行 id 不变（用固定 id 断言）
- 事务回滚：中途抛错后表为空（用 mock statement 抛错）

### V1-3 resolve.ts：引用解析

**目标**：`unresolved_refs` → `edges`，按名匹配 nodes。

**文件**：`packages/core/src/indexdb/resolve.ts`

**API 契约**：
```typescript
export interface ResolveStats {
  resolved: number;
  unresolved: number;
}

export class RefResolver {
  constructor(db: IndexDb);
  /** 全量解析：扫 unresolved_refs，按名匹配 nodes，写 edges。 */
  resolveAll(): ResolveStats;
}
```

**实施细节**：
- **匹配策略**（v1 启发式，按优先级）：
  1. 同文件 + 同语言：`SELECT id FROM nodes WHERE name=? AND file_path=? AND language=?`
  2. 同语言全局：`SELECT id FROM nodes WHERE name=? AND language=?`（取第一个 / PageRank 高的，v1 无 PageRank 取最近定义）
  3. 跨语言：v1 不做（留 SCIP）
- **写 edges**：`kind` 取自 `unresolved_refs.reference_kind`（目前只有 `calls`），`provenance = 'heuristic'`，`line/col` 取自 ref。
- **去重**：同一 `(source, target, kind, line)` 只写一条。
- **不删 unresolved_refs**：解析后保留，供 Dashboard 展示未解析项 + v2 SCIP 重解析用。只标记已解析（可选加 `resolved_node_id` 列，但 schema 没这列——v1 不加，靠 edges 表存在性判断）。

**测试**（`indexdb/resolve.test.ts`）：
- TS 文件 A 调用 B 的函数 → resolve 后 edges 有一条 source=A 的 method id, target=B 的 function id
- 同名歧义（两个文件都有 `format`）→ 优先同文件匹配
- 未匹配的 ref（`bar` 不存在）→ unresolved 计数 +1，无 edge

### V1-4 search.ts：检索管线

**目标**：query → BM25 召回 + co-location 图扩展 → 双路 RRF → top-k。

**文件**：`packages/core/src/indexdb/search.ts`

**API 契约**：
```typescript
export interface SearchParams {
  query: string;
  topK?: number;        // 默认 20
  bm25Weight?: number;  // RRF 用，默认 1
  graphWeight?: number; // RRF 用，默认 1
  graphDepth?: number;  // co-location 扩展深度，默认 1
}

export interface SearchRow {
  chunkId: string;
  filePath: string;
  startLine: number;
  endLine: number;
  body: string;
  scores: { bm25: number; graph: number; rrf: number };
  relatedNodes: string[];
}

export class SearchEngine {
  constructor(db: IndexDb);
  search(params: SearchParams): SearchRow[];
}
```

**实施细节**：
- **BM25 路**（FTS5）：
  ```sql
  SELECT c.id, c.file_path, c.start_line, c.end_line, c.body,
         bm25(chunks_fts) AS score
  FROM chunks_fts
  JOIN chunks c ON c.id = chunks_fts.id
  WHERE chunks_fts MATCH ?
  ORDER BY score
  LIMIT ?
  ```
  - FTS5 的 `bm25()` 返回**负值**（越小越相关），需取负或统一符号。
  - query 预处理：CamelCase 拆分（`TokenBudget` → `Token Budget`）+ 小写，提升召回。抄 CodeGraph `context/index.ts:284` 的边界拆分。
  - MATCH 语法：用空格连接的 AND 查询（`token budget` → `token AND budget` 默认即此行为）。

- **co-location 图路**：
  1. 先用 BM25 取 top-N（N = topK × 3）作为种子 chunk。
  2. 对每个种子 chunk，找**同文件的相邻 chunk**（`start_line` 在 ±窗口内）+ **同 scope 的 symbol 所属 chunk**。
  3. 这些邻居作为 graph 路的命中，rank 按邻近度（同 scope > 同文件相邻 > 同文件）。
  - 无调用图时的近似：co-location 是 CodeGraph 在无 SCIP 时的 fallback 启发式。

- **RRF 融合**：
  ```
  rrf_score(d) = bm25Weight * 1/(60 + rank_bm25(d)) + graphWeight * 1/(60 + rank_graph(d))
  ```
  - 两路各自按 score 排序得 rank（从 1 开始）。
  - 只在 BM25 路出现的文档：graph rank = ∞（该项 = 0）。
  - 只在 graph 路出现的（扩展来的邻居）：bm25 rank = ∞（该项 = 0）。
  - 合并后按 rrf_score 降序，取 topK。

- **relatedNodes**：每个命中 chunk 所在文件的 top symbols（按 symbol 数 / 文件中心性，v1 简单取该文件 symbol 列表前 5 个）。

- **空 query / 极短 query**：< 2 字符直接返回空数组，不走 SQL。

**测试**（`indexdb/search.test.ts`）：
- 建库写入 budget 模块，`search("token budget")` 返回含 `TokenBudget` 类的 chunk，rank=1
- `search("nonexistentterm")` 返回空
- co-location：命中某 chunk 后 relatedNodes 含同文件其他 symbol
- RRF 排序：构造一个 BM25 排第 2 但 graph 排第 1 的 chunk，验证融合后排第 1
- topK 截断：返回数 ≤ topK

### V1-5 Engine 接线

**目标**：把 V1-1..V1-4 接进 Engine，`index()` / `search()` / `close()` 真实可用，`pack()` 可读索引。

**文件**：`packages/core/src/Engine.ts`

**改动**：
1. **构造时开 DB**：`Engine.open()` 里 `new IndexDb({ dbPath })` + `migrate()` + `prepareStatements()`。存到 `this.indexDb`。
2. **`index()`**（替换 stub）：
   ```
   ingest → parseFilesCached → store.writeIncremental(changedPaths)
          → resolve.resolveAll() → 返回 IndexStats（从 store/resolve stats 聚合）
   ```
   - `changedPaths` = parse-cache miss 的文件集合。
   - 增量：parse-cache hit 的文件不写 store（其 nodes/chunks 已在库）。
3. **`search()`**（替换 stub）：`new SearchEngine(this.indexDb).search({query, topK, ...})` → 映射成 `SearchHit[]`。
4. **`close()`**：`this.indexDb.close()`。
5. **`pack()` 读索引**（可选优化，V1-5 内做轻量版）：
   - 检测 `this.indexDb` 是否有数据（`SELECT count(*) FROM files`）。
   - 有数据：从 `nodes` 表读 symbols（替代 parseFilesCached 的 symbols 产出），chunks 从 `chunks` 表读。sources 仍需从磁盘读（pack 要正文）。
   - 无数据：走现有内存管线（不变）。
   - **注意**：v1 先只把 **symbols** 来源切到 SQLite，chunk 正文仍从磁盘读（因为 pack 要完整正文，chunk body 可能被压缩）。完整 pack-on-index 留 v1.5。

**测试**（`Engine.test.ts` 新增）：
- `engine.index()` 后 `IndexStats` 有 symbols/edges/chunks > 0
- `engine.search("token")` 返回非空 hits
- `engine.close()` 后再调方法抛错（连接已关）
- pack 在有索引和无索引下结果一致（token 数接近，允许 skeleton id 因路径差异微变）

### V1-6 CLI cv search

### V1-7 MCP server

**目标**：`@codingverse/mcp` 包，stdio server，暴露 4 个 tools。

**文件**：

**API 契约**（4 tools）：
```typescript

input:  { query: string; topK?: number; projectPath: string }
```

- **错误处理**：tool 异常返回 `{ isError: true, content: [{ type: 'text', text: err.message }] }`，不崩 server。

## 十、实施纪律
