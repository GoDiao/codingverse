# codingverse v2 实施计划

> 目标：在 v1 的 SQLite 索引底座上加**调用图遍历**（callers/callees/impact）+ **Personalized PageRank 排序** + **SCIP 可选接入**，把 v1 的"启发式名匹配 + co-location 图路"升级为"精确符号图 + PageRank 加权"。
> 前置：v1 已完成（132 tests, 4 packages, `cv index`/`cv search`/MCP 可用，node:sqlite + FTS5 trigram + FK-ON + lazy IndexDb + prune）。
> 本计划是 v2 的**单一事实来源**，实施按本文里程碑顺序执行。

---

## 一、v2 范围边界

### 做
1. **调用图遍历** — `Engine.callers(nodeId, depth)` / `callees(nodeId, depth)` / `impact(nodeId, depth)`，读 v1 已有的 `edges` 表做 BFS/反向 BFS。
2. **Personalized PageRank** — 在 `edges` 表上跑迭代 PageRank，回写 `nodes.pagerank` 列；`pack` 的 importance 排序 + `search` 的 relatedNodes 改为读 PageRank。
3. **impact radius 容器下钻** — 改一个类方法 → 追所有 caller，同 depth 下钻到类/文件级（抄 CodeGraph `traversal.ts:456-522`）。
4. **SCIP 可选接入** — `cv index --scip <path/to/index.scip>` 导入 SCIP 索引，产出 `provenance='scip'` 的精确边（替代/增强 v1 的 `provenance='heuristic'` 名匹配边）。
5. **CLI 扩展** — `cv callers` / `cv callees` / `cv impact` / `cv rank`（重算 PageRank）。
6. **MCP 扩展** — 在 v1 的 4 tools 基础上加 `callers` / `callees` / `impact` 3 tools。

### 不做（明确推迟）
| 留给 | 内容 | 代码标记 |
|------|------|---------|
| v2.5 | Dashboard 完整面板（6 板块） | `Engine.stats` stub |
| v3 | YAML rule 组合 + 可编程模板 + llms.txt | — |
| v1.5+ | embedding 向量路（binarized 或真向量） | `chunks.embedding_tokens` 列留空 |
| v1.5 | pack-on-index / store.writeIncremental 接线 | 已实现未接 |

### 已定决策（继承自 v1 + v2 ��增）
1. **PageRank 用纯 JS 迭代**（networkx 风格 power iteration），不引图库（无 `graphology`/`@dagrejs/graphlib`）。节点数 = symbols（v1 demo ~220），迭代收敛快，无需 native。
2. **SCIP 解析用 `scip-typescript` bindings**（SCIP 官方 TS binding，protobuf 解析）。**可选依赖**：只在用户显式 `--scip` 时加载，默认 `cv index` 不依赖它（避免给所有用户加 protobuf 包体积）。
3. **PageRank 触发时机**：`cv rank` 显式跑；`cv index` 后**不自动跑**（避免每次 index 都付迭代成本）。`pack`/`search` 读 `nodes.pagerank`，若全 0（未跑过 rank）则 fallback 到 v1 的 importance 启发式。
4. **impact 下钻策略**：抄 CodeGraph —— BFS 沿 `edges` 反向走，遇到容器符号（class/interface/struct）时下钻到其所有 method 的 caller，同 depth 内展开容器。

## 二、关键技术决策（衍生于已定四条）

| 决策 | 选择 | 理由 / 验证 |
|------|------|------------|
| PageRank 实现 | 纯 JS power iteration，damping=0.85，ε=1e-6，max_iter=100 | Aider 用 networkx `pagerank`；我们无图库依赖，手写 ~40 行。v1 demo 220 节点迭代 <50ms |
| PageRank 节点粒度 | **symbol 级**（nodes 表），非文件级 | Aider 是文件级（节点=rel_fname）；我们已有 symbol 图，symbol 级更精确，且 impact 遍历天然 symbol 级 |
| PageRank 边权重 | `weight = mul * sqrt(num_refs)`（抄 Aider `repomap.py:481-525`） | `mul` 乘子：私有 `_xxx` ×0.1，过热符号（被 >5 文件定义）×0.1，其余 ×1。v2 先实现基础版（无 mentioned_idents/chat 文件概念，那是 aider 的对话场景），personalization 留 v2.5 |
| PageRank 持久化 | 回写 `nodes.pagerank` 列（schema 已有） | `cv rank` 跑完落盘，后续 pack/search 直接读 |
| PageRank fallback | 全 0 时用 v1 importance 启发式 | 不强制用户跑 rank；pack/search 行为不回归 |
| SCIP binding | `scip-typescript`（官方 TS protobuf binding） | 可选依赖，`--scip` 时动态 `import()`，默认不装。验证：`scip-typescript` 在 npm，protobuf 解析纯 JS |
| SCIP 边 provenance | `provenance='scip'` | schema 的 `EdgeProvenance` 已含 `"scip"`（types.ts:172） |
| SCIP 与 heuristic 边共存 | SCIP 导入时**先删同文件 heuristic 边**再插 scip 边 | 避免重复；SCIP 精确度高于 heuristic，优先用 SCIP |
| callers/callees 遍历 | SQL 递归 CTE（`WITH RECURSIVE`）or JS BFS | node:sqlite 支持 `WITH RECURSIVE`；但 depth 控制 + 容器下钻在 JS 里更灵活 → **JS BFS + 批量 SQL 取邻居**（每层一次 `SELECT ... WHERE source IN (...)`） |
| impact 下钻 | BFS 反向 + 容器展开（抄 CodeGraph `traversal.ts:456`） | 遇 class/interface → 展开其所有 method 的 caller，同 depth 内完成 |
| MCP 新 tools | `callers`/`callees`/`impact` 3 个 | 与 CLI 对齐；input `{nodeId, depth?, projectPath}` |

## 四、逐里程碑实施细节

### V2-1 graph.ts：调用图遍历

**目标**：`callers(nodeId, depth)` / `callees(nodeId, depth)` / `impact(nodeId, depth)`，读 `edges` 表做 BFS。

**文件**：`packages/core/src/indexdb/graph.ts`（新建）

**API 契约**：
```typescript
import type { IndexDb } from "./db.js";
import type { SymbolNode } from "@codingverse/shared";

export interface GraphResult {
  nodes: SymbolNode[];
  /** edges traversed, as {source, target, kind, line} for display. */
  edges: { source: string; target: string; kind: string; line: number | null }[];
  /** nodes per depth level, for visualization. */
  byDepth: SymbolNode[][];
}

export class CallGraph {
  constructor(db: IndexDb);
  /** Who calls this node (reverse BFS along edges where target=node). */
  callers(nodeId: string, depth = 1): GraphResult;
  /** What this node calls (forward BFS along edges where source=node). */
  callees(nodeId: string, depth = 1): GraphResult;
  /** Impact radius: reverse BFS with container drill-down (改方法→追所有 caller, 遇 class 下钻). */
  impact(nodeId: string, depth = 3): GraphResult;
}
```

**实施细节**：

1. **SymbolNode 构造**：`edges` 表只存 id，遍历返回需 join `nodes` 拿完整 SymbolNode。每层 BFS 取邻居 id 后，一次 `SELECT * FROM nodes WHERE id IN (...)` 批量取回。准备一个 `nodesById(ids: string[]): SymbolNode[]` helper。

2. **callers(nodeId, depth)** — 反向 BFS：
   - 层 0 = {nodeId}。
   - 每层：`SELECT source FROM edges WHERE target IN (curLayerIds) AND kind = 'calls'` → 下一层 id 集。
   - 去重（已访问的不再入队）。
   - 跑 `depth` 层停止。
   - 收集所有访问到的 node + traversed edges。

3. **callees(nodeId, depth)** — 正向 BFS：同上但 `WHERE source IN (...)`。

4. **impact(nodeId, depth)** — 反向 BFS + 容器下钻（抄 CodeGraph `traversal.ts:456-522`）：
   - 先查 `nodeId` 的 SymbolNode。若它是容器（class/interface/struct/enum）的成员（通过 `qualified_name` 的 `::` scope 判断，或查同文件 scope 包含它的 class），先展开到容器级。
   - 反向 BFS 沿 `calls` 边。
   - **容器下钻**：当某层命中一个 method，且该 method 属于某 class，把同 class 的**所有 method 的 caller** 也纳入（同 depth 内展开，不增加 depth）。
   - depth 默认 3（impact 要看远一点）。
   - 去重。

5. **容器判断**：v1 的 `nodes.qualified_name` 是 `scope::name` 形式（scope 是外层 class 名列表）。一个 method `m` 在 class `C` 里 → `qualified_name = "C::m"`。容器下钻时：取 `m` 的 scope `["C"]` → 找同文件 `qualified_name LIKE "C::%"` 的所有 method → 把它们的 caller 纳入。

6. **prepared statements**：构造器里准备 `selectCallersByTarget`（`SELECT source, line FROM edges WHERE target IN (...) AND kind='calls'` — 注意 IN-list 用动态占位符，或改成 `WHERE target = ?` 循环）、`selectCalleesBySource`、`nodesByIds`、`nodesByQualifiedNamePrefix`。**node:sqlite 不支持变长 IN-list 占位符** → 用循环 `WHERE target = ?` 逐个查，或动态拼 `(?,?,?)`。v1 的 resolve.ts/search.ts 用的是固定占位符；这里层规模小（每层几十个 id），循环查可接受。或者动态拼占位符字符串（`const ph = ids.map(()=>'?').join(',')`）。

7. **空图/无调用**：若 `edges` 表空或 nodeId 无边，返回 `{ nodes: [<nodeId's node>], edges: [], byDepth: [[node]] }`（至少返回起点）。nodeId 不存在 → 抛 `Unknown node id`。

**测试**（`indexdb/graph.test.ts`）：
- **callers 简单**：A calls B, B calls C → `callers(C, 2)` 返回 [B, A]，edges 2 条，byDepth=[[C],[B],[A]]。
- **callees 简单**：同图 → `callees(A, 2)` 返回 [B, C]。
- **depth 截断**：`callers(C, 1)` 只返回 [B]（不返回 A）。
- **impact 容器下钻**：`class X { m() { helper(); } n() { other(); } }`，`helper` 被 `X::m` 调用 → `impact(helper, 2)` 应包含 `X::n` 的 caller（同 class 展开后 `X::n` 的 caller 也纳入）—— **这是 impact 区别于 callers 的核心**。构造一个 `X::n` 被 `caller2` 调用的场景，验证 `caller2` 出现在 impact 结果里。
- **去重**：A→B, A→B（两条边，dedup 后只一条）。
- **无调用**：孤立 node → callers/callees 返回 {nodes:[自身], edges:[], byDepth:[[自身]]}。
- **未知 nodeId** → 抛错。

### V2-2 rank.ts：Personalized PageRank

**目标**：在 `edges` 表上跑 PageRank，回写 `nodes.pagerank`。

**文件**：`packages/core/src/indexdb/rank.ts`（新建）

**API 契约**：
```typescript
export interface RankOptions {
  damping?: number;    // default 0.85
  epsilon?: number;    // convergence threshold, default 1e-6
  maxIter?: number;    // default 100
}

export interface RankStats {
  iterations: number;
  converged: boolean;
  durationMs: number;
  nodeCount: number;
  edgeCount: number;
}

export class PageRank {
  constructor(db: IndexDb);
  /** Compute PageRank over nodes/edges, write back to nodes.pagerank. */
  rank(opts?: RankOptions): RankStats;
  /** Get a node's pagerank (0 if unranked). */
  get(nodeId: string): number;
  /** Top-N nodes by pagerank, for display/relatedNodes. */
  topN(n: number, filePath?: string): { id: string; pagerank: number }[];
}
```

**实施细节**：

1. **构图**（内存）：
   - `SELECT id FROM nodes` → 节点集，建立 `id → index` 映射。
   - `SELECT source, target FROM edges WHERE kind = 'calls'` → 边集。
   - 计算每个节点的 out-degree（`source → count of outgoing calls`）。

2. **边权重**（抄 Aider `repomap.py:481-525` 简化版）：
   - v2 基础版：`weight = 1`（不加 mul 乘子；mentioned_idents/chat 是 aider 对话场景，我们没有）。
   - **私有符号降权**：`name` 以 `_` 开头 → `weight × 0.1`（需要读 `nodes.name`，构图时一起取）。
   - **过热符号降权**：一个 `name` 被 >5 个**不同文件**定义 → `× 0.1`（先扫一遍 nodes 统计 `name → Set<filePath>`，>5 的标记过热）。
   - 权重应用到出边：`edge_weight = mul_source / out_degree_source`（PageRank 标准归一化 + Aider 的 mul 乘子在分子）。

3. **Power iteration**：
   ```
   N = node count
   rank[v] = 1/N  (初始化均匀)
   for iter in 1..maxIter:
     new_rank[v] = (1-d)/N + d * Σ_{u→v} rank[u] * weight(u→v) / out_degree(u)
     // 处理 dangling nodes (出度为0的节点): 其 rank 均匀分给所有节点
     dangling_sum = Σ_{u: out_degree=0} rank[u]
     new_rank[v] += d * dangling_sum / N
     delta = Σ |new_rank[v] - rank[v]|
     rank = new_rank
     if delta < epsilon: converged; break
   ```

4. **回写**：`UPDATE nodes SET pagerank = ? WHERE id = ?`，逐个（或批量 `db.exec` 拼 CASE WHEN — v1 store 用循环 prepared statement，这里同样）。**事务包裹**（BEGIN/COMMIT/ROLLBACK，同 store.ts 模式）。

5. **fallback**：`pack`/`search` 读 `nodes.pagerank`，若**全 0**（未跑过 rank）→ 用 v1 的 importance 启发式（`hasSymbols ? 1000 + symbolCount : 0`）。这个 fallback 在 V2-3 接线时实现，rank.ts 本身只管算+回写。

6. **topN**：`SELECT id, pagerank FROM nodes [WHERE file_path = ?] ORDER BY pagerank DESC LIMIT ?`。用于 `search` 的 relatedNodes（v1 是 `ORDER BY start_line LIMIT 5`，v2 改 `ORDER BY pagerank DESC`，若 pagerank 全 0 则 fallback 到 start_line）。

7. **get(nodeId)**：`SELECT pagerank FROM nodes WHERE id = ?`。

8. **prepared statements**：`selectAllNodeIds`, `selectAllCallEdges`, `updatePagerank`, `topNByPagerank`, `getPagerank`。

**测试**（`indexdb/rank.test.ts`）：
- **收敛**：构造一个小图（3 节点 A→B→C, A→C），`rank()` → `converged=true`, `iterations < maxIter`, `durationMs` 合理。
- **回写**：rank 后 `SELECT pagerank FROM nodes WHERE id='A'` > 0（A 有出度，rank 应非零）。
- **hub 节点高分**：构造一个被多方调用的 "hub" 函数 → 其 pagerank 高于叶子节点。
- **私有降权**：`_private` 的 rank 显著低于 `public_fn`（同结构下）。
- **空图**：无 edges → rank() 仍跑完，所有节点 rank = 1/N（均匀），converged=true（第一次迭代 delta 就 < epsilon）。
- **fallback 信号**：未跑 rank 时 `get(any)` = 0。
- **topN**：rank 后 `topN(3)` 返回按 pagerank desc 的前 3。
- **幂等**：rank() 两次，结果一致。

### V2-3 Engine 接线

**目标**：把 V2-1 graph + V2-2 rank 接进 Engine；pack/search 读 pagerank（带 fallback）。

**文件**：`packages/core/src/Engine.ts`（修改）

**改动**：

1. **替换 stubs**：
   - `callers(nodeId, depth)` → `new CallGraph(this.ensureIndexDb()).callers(nodeId, depth)`，返回 `SymbolNode[]`（Engine API 返回 `SymbolNode[]`，不是 GraphResult —— 检查 `packages/shared` 的 Engine 契约；design.md:264 `callers(nodeId, depth?): Promise<SymbolNode[]>`。所以 Engine 层把 GraphResult.nodes 映射成 `SymbolNode[]`）。
   - `callees(nodeId, depth)` → 同上。
   - `impact(nodeId, depth)` → 同上。

2. **新增 `rank()` 方法**：
   ```typescript
   async rank(opts?: RankOptions): Promise<RankStats> {
     const db = this.ensureIndexDb();
     return new PageRank(db).rank(opts);
   }
   ```

3. **pack 读 pagerank**：
   - v1 的 `compress/index.ts:111` importance = `(r.hasSymbols ? 1000 : 0) + p.symbols.length`。
   - v2 改为：先查 `SELECT AVG(pagerank) FROM nodes WHERE file_path = ?`，若 > 0（已 rank 过）→ importance 用 pagerank 聚合（如 `1000 + avgPagerank * 10000`）；若全 0 → fallback v1 启发式。
   - **最小侵入做法**：在 `compress` 入口加一个可选 `importanceProvider: (path) => number`，Engine.pack 传入一个读 SQLite pagerank 的函数；compress 不直接依赖 IndexDb（保持 layer 分离）。若 provider 返回 0 或未传 → 用 v1 启发式。

4. **search 读 pagerank**：
   - v1 的 `search.ts` relatedNodes = `SELECT id FROM nodes WHERE file_path=? ORDER BY start_line LIMIT 5`。
   - v2 改为 `ORDER BY pagerank DESC, start_line ASC LIMIT 5`（pagerank 全 0 时退化为 start_line 顺序 —— SQL 层自动处理，无需 JS fallback）。
   - search 的 BM25 + co-location + RRF 本身**不改**（pagerank 主要影响 relatedNodes 和 pack 的 layer 选择，不影响 chunk 召回排序 —— v2 不把 pagerank 接进 RRF，那是 v2.5 的"graph 路用 pagerank 加权"）。

5. **close()**：不变（lazy IndexDb 已在 v1 处理）。

**测试**（`Engine.test.ts` 新增）：
- `engine.callers(nodeId, 2)` 返回 SymbolNode[]（非空，含预期 caller）。
- `engine.callees(nodeId, 2)` 同上。
- `engine.impact(nodeId, 3)` 同上 + 验证容器下钻。
- `engine.rank()` → RankStats converged，之后 `engine.callers` 的结果节点 `pagerank > 0`。
- `engine.pack()` 在 rank 后，高 pagerank 文件更可能在 full 层（构造一个 hub 文件，rank 后 pack 低预算，验证 hub 文件比叶子文件更可能 full）。
- **fallback**：未 rank 时 pack 仍正常工作（用 v1 启发式，不回归）。
- 未知 nodeId → callers/callees/impact 抛错。

### V2-4 CLI 扩展

**目标**：`cv callers` / `cv callees` / `cv impact` / `cv rank`。

**文件**：`packages/cli/src/commands/{callers,callees,impact,rank}.ts`（新建）+ `bin.ts` 注册。

**命令规格**：
```
cv callers <nodeId> [path] [-d 1]          # who calls this node
cv callees <nodeId> [path] [-d 1]          # what this node calls
cv impact  <nodeId> [path] [-d 3]          # impact radius
cv rank    [path] [--damping 0.85] [--max-iter 100]   # compute PageRank
```

**输出格式**（callers/callees/impact）：
- 每行一个 SymbolNode：`filePath:startLine-endLine  kind  qualifiedName  [pagerank=0.0123]`
- 按 depth 分组输出（`--- depth 1 ---` / `--- depth 2 ---`）。
- stderr 汇总：`[cv callers] 5 nodes in 3ms (depth 2)`。

**cv rank 输出**：
- stderr：`[cv rank] 220 nodes, 279 edges, converged in 23 iters / 45ms`。
- stdout：top 20 节点 by pagerank（让用户看 rank 分布）。

**nodeId 查找辅助**：用户不一定知道 nodeId。加 `cv rank --find <name>` → 模糊查 nodes.name → 打印匹配的 id（方便复制到 callers/callees）。或者 `cv callers <name>` 自动按 name 解析到 id（取第一个匹配）。**v2 简化**：`cv callers` 接受 name 或 id；若是 name，`SELECT id FROM nodes WHERE name = ? LIMIT 1` 解析。

**测试**：无单元测试（CLI 约定），E2E 手动验证（见各里程碑验收）。

### V2-5 SCIP 可选接入

**目标**：`cv index --scip <path.scip>` 导入 SCIP 索引，产出精确边。

**文件**：
- `packages/core/src/indexdb/scip.ts`（新建）— SCIP 解析 + 边导入
- `packages/cli/src/commands/index.ts`（修改）— 加 `--scip` 选项
- `packages/core/package.json` — `scip-typescript` 作为 **optionalDependencies**（或运行时动态 `import()`，不进 dependencies）

**API 契约**（scip.ts）：
```typescript
export interface ScipImportOptions {
  /** Path to .scip index file. */
  scipPath: string;
  /** Repo root (for resolving relative paths in SCIP to our file_path). */
  repoRoot: string;
}

export interface ScipImportStats {
  documents: number;
  occurrences: number;
  relationships: number;
  edgesInserted: number;
  edgesReplaced: number;  // heuristic edges replaced by scip
}

export class ScipImporter {
  constructor(db: IndexDb);
  import(opts: ScipImportOptions): Promise<ScipImportStats>;
}
```

**实施细节**：

1. **scip-typescript 动态加载**：
   ```typescript
   const { Index } = await import("scip-typescript");  // runtime dynamic
   ```
   - 不在文件顶 import，避免静态依赖。`scip-typescript` 加进 `optionalDependencies`（pnpm install 默认装，但失败不阻塞；或完全不进 deps，用户 `pnpm add -g scip-typescript` 或项目内按需装）。
   - **决策**：加进 core 的 `optionalDependencies`，运行时 `import()` 失败时给清晰错误：`"SCIP support requires scip-typescript. Install: pnpm --filter @codingverse/core add scip-typescript"`。

2. **解析 .scip**：
   - `const buf = fs.readFileSync(scipPath); const index = Index.deserialize(buf);`（scip-typescript API — 验证实际 API）。
   - 遍历 `index.documents`：每个 document 的 `relative_path` → 映射到我们的 `file_path`（相对 repoRoot）。
   - 遍历 `document.occurrences`：`symbol_roles` 含 Definition 的 → 对应我们的 node（按 `symbol` 字符串匹配 `nodes.qualified_name` 或 `nodes.name` —— SCIP symbol 是 `scheme package descriptor+` 格式，需要解析 descriptor 取最后一段 name + 拼 qualified_name）。**匹配策略**：SCIP symbol 末段 name → 在 `nodes` 里按 `name + file_path` 找；找不到跳过（SCIP 可能覆盖我们没解析的文件）。
   - 遍历 `document.symbols[].relationships`：`is_reference` → 一条 edge（source = enclosing function's symbol, target = relationship.symbol 对应的 node, kind = 'calls' 或 'references'）。
   - **enclosing function 解析**：SCIP occurrence 有 `enclosing_range` → 找该 range 内的 definition occurrence → 其 symbol → 对应 node。这是 caller 的 node id。

3. **边导入**：
   - `provenance = 'scip'`。
   - **替换策略**：导入某文件的 SCIP 边前，`DELETE FROM edges WHERE source IN (SELECT id FROM nodes WHERE file_path = ?) AND provenance = 'heuristic'`（清掉该文件的 heuristic 边），再插 scip 边。避免重复 + SCIP 优先。
   - 事务包裹。
   - 去重（同 source/target/kind/line 只一条）。

4. **CLI `--scip`**：
   - `cv index [path] --scip <path.scip>` → 先正常 `engine.index()`（建 heuristic 边），再 `ScipImporter.import({scipPath, repoRoot})` 替换。
   - 或 `cv index --scip-only <path.scip>` → 只导 SCIP 不跑 heuristic（适合已 index 过的仓库增量补 SCIP）。

5. **未 index 的仓库 + SCIP**：SCIPImporter 假设 `nodes` 表已有数据（v1 index 先跑）。若 nodes 空 → 抛错 "Run `cv index` before `--scip`"。

**测试**（`indexdb/scip.test.ts`）：
- **需构造测试用 .scip 文件**：用 scip-typescript 的 API 反向构造一个 minimal Index（或 fixture 一个小 .scip 二进制）。如果构造困难，**skip 单元测试**，靠 E2E 验证（用真实 scip-typescript indexer 生成一个 .scip）。**决策**：若 scip-typescript 提供 `Index` 构造 API → 单元测试用构造的 Index；否则 skip 单元测试，V2-5 靠 E2E + typecheck 验收。
- **E2E**（需真实 .scip）：用 `scip-typescript` indexer 对 codingverse 自己生成 `index.scip` → `cv index --scip index.scip` → 验证 `edges` 表有 `provenance='scip'` 的边，且数量 > 0。

## 八、风险与回退

| 风险 | 概率 | 回退 |
|------|------|------|
| PageRank 不收敛（环形图 / 孤岛） | 低 | damping=0.85 + dangling 均匀分配保证收敛；maxIter 兜底；测验证 |
| impact 容器下钻复杂度爆炸 | 中 | 限制同 depth 展开数（如 max 50 节点/层），超限截断 + 标记 |
| SCIP symbol 匹配率低（SCIP symbol 格式 vs 我们的 qualified_name） | 中 | 按末段 name + file_path 匹配；匹配率 <50% 时 warn 不报错；v2.5 再优化 symbol 映射 |
| scip-typescript API 变化 / 包不可用 | 中 | optionalDependencies + 动态 import + 清晰错误；用户可 skip SCIP，v2 核心功能（callers/rank/impact）不依赖 SCIP |
| PageRank 回写大表慢（万级节点） | 低 | v1 demo 220 节点 <50ms；万级用批量 UPDATE + 事务；超大规模留 v2.5 优化 |
| pack importance 改动破坏 v1 pack | 中 | fallback 机制 + 回归测试（未 rank 时 v1 行为不变） |
| BFS 深查询慢（depth 大 + 图密） | 低 | depth 默认小（callers/callees=1, impact=3）；加 LIMIT + 去重 |

## 十、实施纪律

1. **严格按里程碑顺序**：V2-1 不绿不进 V2-2（graph 是 rank/impact 的基础）。
2. **每里程碑一个 commit**：`v2-Vn: <描述>`。
3. **TDD**：graph/rank 的 .test.ts 先行；scip 若无法 fixture 则 E2E 验收。
4. **不改 schema**：v2 复用 v1 的 `nodes.pagerank`/`edges.provenance` 列，不动 schema.sql（除非 V2-1 发现需要补充索引，且只在 `db.ts migrate()` 加 `IF NOT EXISTS` 索引）。
5. **不越界**：Dashboard（v2.5）、embedding 向量（v1.5+）、YAML rule（v3）碰到就停，记 TODO。
6. **SCIP 可选**：不装 scip-typescript 时所有 v2 功能（callers/rank/impact）正常；`--scip` 才需依赖。
7. **fallback 优先**：pack/search 的 pagerank 集成必须有 fallback，未 rank 不回归 v1 行为。
8. **C 盘保护**：scip-typescript 走 pnpm store（E 盘）；无全局安装。
9. **subagent 驱动**：用 v1 验证过的 subagent-driven development 模式（implementer + spec review + code quality review per milestone），每里程碑两阶段评审。
