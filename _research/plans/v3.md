# codingverse v3 实施计划

> 目标：把"索引一次、三模出口"从**能用**推进到**agent 真正好用** —— 让打包出口从"整仓快照"进化成"按需组装"（变更范围 / 检索驱动），并补齐检索的诚实性（真调用图并入融合）与索引的活性（watch 持续索引）。
> 前置：v2.5 完成（Dashboard 6 板块 + `/api/pack-content` 打包出口 + 节点详情 + 图谱 pan/zoom；`cv index/search/pack/rank/callers/callees/impact/expand/status/serve` + MCP 7 tools 全可用）。
> 本计划是 v3 的**单一事实来源**，实施按里程碑顺序执行。

---

## 一、v3 范围边界

### 定稿决策（本次确认）
1. **向量 / embedding 检索：正式移出路线图**（不再是 v1.5 待办）。理由沿用既定判断——binarized 精度损失、真向量引重依赖，而句法级 BM25+图谱对"代码"这种强结构文本已足够；语义模糊召回不是本工具的价值点。`chunks.embedding` / `embedding_tokens` 列保留为 schema 预留位（NULL），不删、不接线。
2. **llms.txt 生成：降级为"可选糖"，默认不做**。原因：board ⑤ 打包出口（预览 + Copy + Download + xml/md/json）已覆盖"把仓库导成 LLM 可读入口"的核心诉求；llms.txt 只是另一种清单格式，价值增量低。仅在 V3-6 作为可选项列出。
3. **可编程模板（Handlebars）：降级为"可选糖"**。三格式（xml/md/json）已覆盖绝大多数消费方；自定义模板是长尾需求，放 V3-6 可选。

### 做（按价值/成本排序）
1. **V3-1 变更范围打包（diff-scoped pack）** ⭐ —— `cv pack --changed` / `--since <ref>`：只打包变更文件 + 其 impact 半径（callers/被影响者），产出"这次改动 + 所有受影响面"的精准上下文。零新造轮子（复用 blob-hash 变更列表 + `impact` 遍历 + `pack`）。
2. **V3-2 检索驱动打包（query-scoped pack）** ⭐ —— `cv pack --query "<自然语言/符号>"`：search → top-k 命中 → 沿调用图扩邻域 → 按预算分层打包。首次让"检索"与"打包"两个出口**组合**（此前共享索引但从不互相调用）。
3. **V3-3 真调用图并入检索融合** —— 把 search 里的"graph"路从**同文件行邻近**（v1 简化占位）换成**真调用图扩展**（callers/callees，edges 表已有，board ③ 已在用）。修正 v1 起就存在的"co-location ≠ 真图谱"诚实性缺口。
4. **V3-4 watch 持续索引** —— `cv watch`：文件变更增量重索引（复用 blob-hash 横切），保持 `serve`/MCP 的索引常热，避免每次手动 `cv index`。
5. **V3-5 多语言扩展** —— 在声明式 per-language extractor 上新增 Go / Rust / Java（现仅 TS/JS/JSX + Python）。每语言 = 一个 extractor 配置 + tags 查询 + 注册一行。

### 不做（明确推迟或放弃）
| 处置 | 内容 |
|------|------|
| 放弃 | embedding 向量路（binarized / sqlite-vec）—— 本次正式移出 |
| V3-6 可选糖 | Handlebars 可编程模板；llms.txt 生成 |
| 保留推迟 | YAML rule 递归组合（llm-context 路线）—— 需求未验证，观望 |
| 保留推迟 | 跨文件类型推断 —— 无底洞，深度场景仍走 SCIP 接入 |

## 二、关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 变更来源 | `git` blob-hash diff（复用横切 B） | 已有增量缓存的变更列表,零新增采集;`--since <ref>` 走 `git diff --name-only <ref>` |
| impact 扩展深度 | 默认 depth=2,`--depth` 可调 | 与 `cv impact` 一致;depth 过大→上下文爆炸 |
| query→pack 桥接 | 复用 `Engine.search()` + `callersGraph/calleesGraph`,不新写检索 | 两出口组合而非重造 |
| 真图谱融合 | 复用 edges 表 + CallGraph 遍历,RRF 权重/公式不变 | 只换"graph 路"的召回源,融合层不动,风险最小 |
| watch 实现 | `node:fs` `watch`(递归)+ 防抖批量,不引 chokidar | 零依赖,C 盘友好;仅本地开发工具够用 |
| 新语言 | web-tree-sitter 对应 grammar wasm + tags.scm | 声明式 registry 已支持,增量成本低 |
| 模板/llms.txt | 若做,模板用 Handlebars(生态成熟),llms.txt 复用 render 层加一格式 | 仅可选糖,不阻塞主线 |

## 三、逐里程碑实施细节

### V3-1 变更范围打包(diff-scoped pack) ⭐
- **CLI**:`cv pack --changed`(对比工作区 vs HEAD)/ `--since <ref>`(对比任意 ref)。与现有 `--budget/--strategy/--format/-o` 组合。
- **管线**:变更文件集 → 每个文件的符号 → `impact`/`callers` 扩到受影响节点 → 这些节点所在文件进入打包候选 → 正常分层压缩。
- **Engine**:`packScoped({ changedFiles?, since?, depth, ...PackOptions })`,内部求"变更文件 ∪ 受影响文件",其余走 `pack()`。
- **出口**:CLI + MCP 新增 `pack_changed` tool(可选)+ 后端 `/api/pack-content` 加 `scope=changed&since=` 参数,Dashboard board ⑤ 加一个"仅变更"开关。
- **验收**:改一个高扇入文件 → `--changed` 产出应含该文件(full)+ 其 caller(至少 skeleton);无变更时产出空/提示。

### V3-2 检索驱动打包(query-scoped pack) ⭐
- **CLI**:`cv pack --query "<text>"` [`--top-k N`]。
- **管线**:`search(query, topK)` → 命中 chunk 所属节点 → `calleesGraph`/`callersGraph` 扩一跳邻域 → 邻域文件集 → 分层打包(命中文件优先 full)。
- **Engine**:`packQuery(query, { topK, depth, ...PackOptions })`。
- **出口**:CLI + MCP `pack_query` tool + Dashboard board ⑤ 加 query 输入(与 board ④ 检索检查器呼应)。
- **验收**:`--query "token budget"` 产出应把 budget 相关文件排在 full 层,不相关文件降级/omit。

### V3-3 真调用图并入检索融合
- **现状**:`search.ts` 的 graph 路是"同文件行邻近"(`v1 simplified — NOT a real call graph`)。
- **改动**:graph 路召回源换成 `CallGraph` 的 caller/callee 邻接(seed = BM25 命中节点,扩 depth=1~2),proximity 语义从"行距"变成"图距/跳数"。RRF 融合公式、权重、`rrfK` 不变。
- **影响面**:`SearchDebug` 的 co-location 列语义更新(UI 文案:"graph proximity"→"call-graph hops")。board ④ 随之更新说明。
- **验收**:一个被多处调用的函数,query 命中后 graph 路应召回其 caller(跨文件),而非仅同文件邻居。

### V3-4 watch 持续索引
- **CLI**:`cv watch [path]`,可与 `cv serve` 组合(serve 内部起 watch,索引常热)。
- **实现**:`fs.watch` 递归监听(应用 ingest 的 ignore 规则)→ 防抖(默认 300ms)批量 → 对变更文件走增量 `sync()`(blob-hash 命中则跳过)→ 更新 SyncState(board ⑥ 实时反映)。
- **验收**:watch 运行中改一个文件 → 数秒内 board ⑥ 变更列表/耗时更新,`search` 立即可查到新符号。

### V3-5 多语言扩展(Go / Rust / Java)
- 每语言:引入对应 `tree-sitter-<lang>.wasm` + 写 `<lang>.ts` extractor(tags 查询 + capture 映射)+ registry 注册。
- **验收**:各语言样例仓 `cv index` 后,符号数 > 0、能被 search/pack/graph 正确识别;degraded 率在阈值内。

### V3-6 可选糖(按需,不阻塞)
- **Handlebars 模板**:`--template <.hbs>`,render 层加模板分支。
- **llms.txt 生成**:`cv pack --format llms-txt`,输出索引层(清单 + 链接)+ Optional 段(预算闸门)。
- 二者均为可选,主线 V3-1~V3-5 完成后视需要再做。

## 八、实施纪律
1. **严格按里程碑顺序**:V3-1 → V3-2 → V3-3 → V3-4 → V3-5;V3-6 可选,最后。
2. **每里程碑一个 commit**:`v3-Vn: <描述>`。
3. **两出口组合优先于新造**:V3-1/V3-2 只允许复用 impact/search/pack,不新写检索或压缩逻辑。
4. **不越界**:embedding 向量(已弃)、YAML rule(观望)、跨文件类型推断(SCIP)碰到就停,记 TODO。
5. **诚实性优先**:V3-3 必须真正替换 co-location 占位,不允许留"仍是行邻近"的伪图谱。
