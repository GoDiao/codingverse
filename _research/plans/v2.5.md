# codingverse v2.5 实施计划

> 目标：落 Dashboard（三模态第三模「观测」）+ 扫清 v2 polish 遗留的性能项。
> 前置：v2 + polish 完成（207 tests，`cv index`/`search`/`rank`/`callers`/`callees`/`impact`/`expand`/`pack` + MCP 7 tools + SCIP 可选接入全部可用，node:sqlite + FTS5 trigram + FK-ON + lazy IndexDb + 瞬态只读 pack + MAX pagerank 聚合）。
> 本计划是 v2.5 的**单一事实来源**，实施按本文里程碑顺序执行。

## 一、v2.5 范围边界

### 做
1. **`Engine.stats()`** — 实现 stub，返回 `DashboardStats`（已有类型，shared/types.ts:316）：index 概览 + 健康度 + 语言分布 + tokenMap treemap + syncQueue。
2. **CLI `cv status` 扩展** — 从当前的 token treemap 扩到完整索引概览（板块① CLI 版）+ 健康度 + 语言分布。
3. **HTTP server + 单页前端** — `@codingverse/dashboard` 新包，零构建 SPA（单 HTML + 轻量 JS），6 板块可视化。复用 Engine 同进程 SQLite 连接。
4. **`cv serve` 命令** — 启动 Dashboard HTTP server。
5. **6 板块**：
   - ① 索引概览（Engine.stats 数据，统计卡片 + 语言分布饼图）
   - ② Token 地图（treemap，复用 `buildTokenTreemap`）
   - ③ 代码图谱（force-directed 符号图 + PageRank 热度着色 + 调用链高亮）
   - ④ 检索检查器（query 输入 → BM25/co-location 各路命中对比 + RRF 融合过程 + top-k）
   - ⑤ 打包预览（layer 选择预览 + 每文件层级标注 + token 估算）
   - ⑥ 同步状态（parse-cache hit/miss + git blob hash 变更 + 上次索引耗时）
6. **v2 性能 polish**（顺手扫清）：rank writeback 批量、graph 主 BFS 上限、SCIP per-occurrence SQL、per-call 构造缓存、未注册 statement。

### 不做（明确推迟）
| 留给 | 内容 |
|------|------|
| v3 | YAML rule 组合 + 可编程模板 + llms.txt |
| v1.5+ | embedding 向量路（binarized 或真向量）— `chunks.embedding_tokens` 仍 NULL |
| v1.5 | pack-on-index / store.writeIncremental 接线 |
| v2.5+ | SCIP `kind='calls'` 拆分（用 `symbol_roles` 区分 calls vs references）— 功能非 polish |

### 已定决策
1. **前端零构建**：单 HTML + 原生 ES modules + D3（treemap/force）from CDN。对标 Repomix `browser/`。不引 React/Vue/构建链。
2. **后端轻量 HTTP**：Node 内置 `node:http`（不引 express/hono）。够用、零依赖、C 盘友好。
3. **WebSocket 用 SSE 替代**：运行时事件（索引进度/检索日志）用 Server-Sent Events（SSE，`text/event-stream`）推送——比 WS 简单，单向推送够用。
4. **Dashboard 复用 Engine**：HTTP handler 调 `Engine.stats()`/`search()`/`callers()` 等，不另起索引进程。一个 `cv serve` = 一个 Engine 实例服务所有请求。
5. **6 板块按数据源就绪度排序**：②（tokenMap 已有）+ ①（stats 实现后）+ ⑥（syncQueue 需 Engine 追踪运行时状态）先做；③（图谱）+ ④（检索检查器）+ ⑤（打包预览）后做。前 3 板块数据全在索引里，后 3 需要额外查询/管线接出。

## 二、关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| HTTP server | `node:http` 内置 | 零依赖，C 盘友好，够用（本地调试工具） |
| 前端框架 | 无框架，原生 ES modules + D3 CDN | 零构建，单 HTML 文件，对标 Repomix `browser/` |
| 图谱可视化 | D3-force（CDN） | 轻量，force-directed 符号图；Cytoscape 重，不引 |
| treemap | D3-treemap（CDN） | 板块② 已有 `buildTokenTreemap` 数据，前端渲染 |
| 实时推送 | SSE（`text/event-stream`） | 比 WS 简单，单向推送够用；node:http 原生支持 |
| 端口 | 默认 7331（`cv serve --port 7331`） | 避开常见端口；可配置 |
| 静态资源 | 内联进 server.ts OR 独立 `web/` 目录 | 决策：独立 `packages/dashboard/web/` 目录，server 读文件返回；避免 server.ts 巨大 |
| Engine 共享 | `cv serve` 持有一个 Engine 实例 | 所有 HTTP handler 共享同一 Engine + SQLite 连接 |
| DashboardStats 扩展 | 已有类型够用板块①②⑥；③④⑤ 需新 query endpoint | 不改 shared 类型，新 endpoint 返回专用 JSON |
| 检索检查器数据 | 新 endpoint `/api/search-debug?query=...` 返回 BM25 + co-location 各路独立命中 + RRF 过程 | V2-4 search.ts 内部有 bm25Rows + graphCandidates + rrf，需拆出中间结果 |
| 打包预览数据 | 新 endpoint `/api/pack-preview?budget=...` 返回 layerMap + 每文件 tokens + layer | V2 compress 结果已有 layerMap/files，直接返回 |
| 图谱数据 | 新 endpoint `/api/graph?depth=...` 返回 nodes + edges 子集 | 大图截断（如 top 200 节点 by pagerank + 其间 edges） |

## 四、逐里程碑实施细节

### V2.5-1 Engine.stats() 实现

**目标**：替换 `Engine.stats()` stub，返回完整 `DashboardStats`。

**文件**：`packages/core/src/Engine.ts`（修改）

**DashboardStats 字段填充**（shared/types.ts:316）：
```typescript
{
  index: {
    files: SELECT COUNT(*) FROM files,
    symbols: SELECT COUNT(*) FROM nodes,
    edges: SELECT COUNT(*) FROM edges,
    chunks: SELECT COUNT(*) FROM chunks,
    dbSize: fs.statSync(index.db).size,
    lastSync: SELECT MAX(indexed_at) FROM files,
  },
  health: { ok: SELECT COUNT(*) FROM files WHERE parse_status='ok', degraded/failed/skipped 同理 },
  languages: SELECT language, COUNT(*) FROM files GROUP BY language → Record<string, number>,
  tokenMap: buildTokenTreemap(...) — 复用 budget/index.ts 的 buildTokenTreemap，数据来自 chunks.token_count 按文件聚合,
  syncQueue: [] — V2.5-4 填充运行时状态，先空数组,
}
```

**实施细节**：
- `Engine.stats()` 调 `ensureIndexDb()`（stats 需要索引）。
- index 概览：5 个 COUNT 查询 + `fs.statSync(indexPath).size` + `MAX(indexed_at)`。
- health：4 个 COUNT WHERE parse_status=...。
- languages：`SELECT language, COUNT(*) AS n FROM files GROUP BY language` → Record。
- tokenMap：**关键**——v1 的 `buildTokenTreemap` 接收 `FileTokenCount[]`（path + tokens）。v2.5 从 `chunks` 表查 `SELECT file_path, SUM(token_count) AS tokens FROM chunks GROUP BY file_path` → 转成 FileTokenCount[] → `buildTokenTreemap`。注意 chunks.token_count 可能为 NULL（v1 未 count 的 chunk）——用 `COALESCE(token_count, 0)` 或先 count 再查。**最简**：直接复用 `Engine.tokenReport()`（v1 已实现，ingest→parse→budget.countFiles→treemap），但那会重新 parse。**决策**：从 SQLite 查（已索引的数据），不重新 parse。若 chunks.token_count 大量 NULL，fallback 调 tokenReport（但那破坏"读索引"原则）。**更好**：V2.5-1 顺带在 store.ts 写 chunks 时填 token_count（v1 store 写了 `chunk.tokenCount ?? null`——parse 时 chunker 没算 token）。补一个 `TokenBudget.countChunk` 在 store 写入时填，使 token_count 非 NULL。这是 V2.5-1 的隐藏前置。**决策**：V2.5-1 加一个"store 写 chunks 时 count token"的小改动（用 TokenBudget），让 tokenMap 能从索引直读。
- syncQueue：先返回 `[]`，V2.5-4 实现。

**测试**（Engine.test.ts）：
- `engine.index()` 后 `engine.stats()` 返回 DashboardStats，各字段非零/正确。
- `stats()` 在未 index 时 → 抛错或返回空（决策：抛 "Run cv index first"，因为 stats 需要索引数据）。
- tokenMap 结构正确（TreemapNode 树）。
- languages Record 正确（TS/Python 等）。
- health 各状态计数正确（构造一个 degraded 文件验证）。

### V2.5-5 板块③ 代码图谱

**目标**：force-directed 符号关系图 + PageRank 热度着色 + 调用链高亮。

**文件**：`packages/dashboard/src/server.ts`（新 endpoint）+ `web/`（板块③ UI）

**新 endpoint**：`GET /api/graph?limit=200` →
```typescript
{
  nodes: [{ id, name, qualifiedName?, filePath, pagerank, kind }] // top N by pagerank,
  edges: [{ source, target, kind }] // edges among the selected nodes only,
  maxPagerank: number, // for color scale
}
```

**板块③ UI**：D3-force（CDN），节点大小/颜色按 pagerank（热度着色），边按 kind 着色（calls 实线 / references 虚线）。点击节点 → 高亮其 callers/callees（调 `/api/callers?id=...` 复用 Engine.callersGraph）。

### V2.5-6 板块④ 检索检查器

**目标**：query → BM25 / co-location 各路独立命中对比 + RRF 融合过程 + top-k。

**文件**：`packages/core/src/indexdb/search.ts`（拆出中间结果）+ `server.ts`（endpoint）+ `web/`（UI）

**新 endpoint**：`GET /api/search-debug?query=...&topK=20` →
```typescript
{
  query: string,
  bm25: [{ chunkId, filePath, startLine, score, rank }],       // BM25 路独立命中
  graph: [{ chunkId, filePath, startLine, proximity, rank }],  // co-location 路独立命中
  fused: [{ chunkId, filePath, startLine, rrf, bm25Rank, graphRank }], // RRF 融合后 top-k
  rrfK: 60,
  weights: { bm25, graph },
}
```

**实施**：search.ts 的 `search()` 内部已有 bm25Rows + graphCandidates + rrf 计算。**决策**：加一个 `searchDebug(params): SearchDebugResult` 方法（或 export 一个拆出中间步骤的函数），返回三路独立 + 融合结果。不破坏现有 `search()`（它返回最终 SearchRow[]）。

**板块④ UI**：query 输入框 → 三列对比（BM25 / co-location / Fused），每列 top-k 列表 + 分数条；点击命中 → 显示 chunk body 预览。RRF 公式可视化（`1/(60+rank)` 各项相加）。

**测试**：search.test.ts 加 `searchDebug` 测试（三路独立命中 + 融合排序）。

### V2.5-8 v2 性能 polish 扫尾

**目标**：扫清 v2 遗留的性能项（非功能变更，纯优化）。

**项**：
1. **rank writeback 批量**：N 条 UPDATE → temp table + JOIN 或 `CASE WHEN` 批量。万级节点加速。
2. **graph 主 BFS 上限**：主 BFS（非容器下钻）加 cap（如 max 200 节点/层），超限截断 + `truncated` flag。
3. **SCIP per-occurrence SQL**：预加载 `Map<name, nodeId>` 一次，避免每个 ref 一次 SQL。
4. **per-call 构造缓存**：`new CallGraph(db)`/`new SearchEngine(db)` 每次 ensureIndexDb 后缓存到 Engine 实例。
5. **未注册 statement**：pack 的 pagerank provider prepared statement 进 `IndexDb.statements` Map。

**测试**：现有测试保绿（无行为变更）；加性能基准测试（可选，v2.5 决策：不加，靠手动观测）。

## 十、实施纪律

1. **严格按里程碑顺序**：V2.5-1（stats 数据基础）不绿不进 V2.5-3（前端依赖数据）。
2. **每里程碑一个 commit**：`v2.5-Vn: <描述>`。
3. **TDD**：core 部分必有单元测试；前端手动验证；endpoint 抽离可测部分。
4. **不越界**：v3（YAML rule/模板）、embedding 向量、pack-on-index 碰到就停，记 TODO。
5. **零构建前端**：不引 React/Vue/构建链；D3 走 CDN；单 HTML + ES modules。
6. **node:http 零依赖**：不引 express/hono；C 盘友好。
7. **性能优化无行为变更**：V2.5-8 每项独立 commit + 回归测试保绿。
8. **C 盘保护**：无全局安装；D3 CDN 不占本地空间；dashboard 包在 repo 内。
9. **subagent 驱动**：沿用 v1/v2 验证过的模式（implementer + spec review + code quality review per milestone），前端里程碑可合并 review（手动验证为主）。
